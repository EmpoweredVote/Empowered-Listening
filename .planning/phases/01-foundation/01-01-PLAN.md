**File 1: `supabase/migrations/20260420000000_create_listening_schema.sql`**

Structure:
1. `CREATE SCHEMA IF NOT EXISTS listening;`
2. In dependency order (parents before children), create all 9 tables:
   - `listening.debates` (from architecture doc §4.1)
   - `listening.debate_speakers` (references debates)
   - `listening.debate_segments` (references debates + debate_speakers)
   - `listening.transcript_entries` (references debates + debate_segments + debate_speakers)
   - `listening.notes` (references debates + debate_segments)
   - `listening.cx_wrap_up_events` (references debates + debate_segments + debate_speakers)
   - `listening.votes` (references debates + debate_segments)
   - `listening.topic_proposals` (references debates)
   - `listening.speaker_performance` (references debates)
3. Create the indexes specified in the architecture doc:
   - `CREATE INDEX ON listening.transcript_entries(debate_id, spoken_at);`
   - `CREATE INDEX ON listening.transcript_entries USING gin (to_tsvector('english', text));`
4. Create the partial unique index to prevent duplicate null-target votes (noted in the architecture doc comment block on `votes`):
   ```sql
   CREATE UNIQUE INDEX votes_null_target_unique
     ON listening.votes (debate_id, voter_id, vote_type)
     WHERE target_id IS NULL;
   ```

All `user_id`, `voter_id`, `created_by`, `proposed_by`, `edited_by` columns reference `auth.users(id)` — this matches the architecture doc.  Note that the onboarding doc recommends referencing `public.users(id)` in some places; where the architecture doc and onboarding doc conflict, follow the architecture doc for v1 (auth.users) because all table definitions are written against it.  Record this choice in the migration file as an SQL comment at the top.

**File 2: `supabase/migrations/20260420000001_listening_rls_policies.sql`**

Structure:
1. For every table in the listening schema, `ALTER TABLE listening.<table> ENABLE ROW LEVEL SECURITY;`
2. Policies per the architecture doc §4 "RLS rules":
   - `debates`: `SELECT` to `authenticated, anon` where `status IN ('live', 'completed')`.  Moderator `UPDATE` is deferred — gate in application code via role check for now (role slugs are Phase 2).
   - `debate_segments`: `SELECT` to `authenticated, anon` when the parent debate is live or completed.  `UPDATE` deferred.
   - `transcript_entries`: `SELECT` to `authenticated, anon` when the parent debate is live or completed.
   - `votes`: `SELECT` aggregates only — for v1 write a simple `SELECT` policy that allows reading own votes: `USING ((select auth.uid()) = voter_id)`.  Tally aggregation queries will use service role (bypasses RLS).  `INSERT` to `authenticated` with `WITH CHECK ((select auth.uid()) = voter_id)`.
   - `notes`: `SELECT` to `authenticated` with `USING ((select auth.uid()) = user_id OR is_private = false)`.  `INSERT` to `authenticated` with `WITH CHECK ((select auth.uid()) = user_id)`.  `UPDATE`/`DELETE` only own notes.
   - `topic_proposals`: `SELECT` to `authenticated, anon`.  `INSERT`/`UPDATE` deferred to application code (moderator check).
   - `debate_speakers`, `speaker_performance`, `cx_wrap_up_events`: `SELECT` to `authenticated, anon`.  All writes go through service role (`pool.query()` with pooler URL) — no public write policies.

Follow the performance rule from the research doc: always include `TO authenticated` or `TO anon` in every policy; always wrap `auth.uid()` as `(select auth.uid())` for index use.

Do NOT set `SUPABASE_JWT_SECRET` anywhere.  Do NOT grant any `TO public` policies.

Rationale: Service role bypasses RLS, so writes from the backend via `pool.query()` work regardless of policies.  RLS only matters if PostgREST is ever exposed later or if client-side Supabase JS is used against `public` views.  For v1 it is defense in depth.
