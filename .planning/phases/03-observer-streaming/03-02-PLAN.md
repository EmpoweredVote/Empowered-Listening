<task type="auto">
  <name>Task 2: Create HlsPlayer client component with Safari fallback and LIVE-delayed badge</name>
  <files>app/debates/[debateId]/HlsPlayer.tsx</files>
  <action>
    Create `app/debates/[debateId]/HlsPlayer.tsx` as a client component.  Make the directory if it does not exist (`mkdir -p "app/debates/[debateId]"`).

    The file must start with `'use client';`.  Import `useEffect, useRef, useState` from react and the default export from `hls.js` as `Hls`.

    Component signature:

    ```ts
    interface HlsPlayerProps {
      src: string;                       // full m3u8 URL — parent is responsible for building it
      className?: string;                // optional extra classes for the outer wrapper
      badgeLabel?: string;               // defaults to 'LIVE · delayed ~5-10s'; 03-04/03-05 can override for VOD
    }
    type PlayerState = 'loading' | 'playing' | 'error';
    ```

    Implementation — follow the exact pattern from `.planning/phases/03-observer-streaming/03-RESEARCH.md` Code Example "Correct hls.js React Client Component":

    1. useRef<HTMLVideoElement>(null) for the <video>.
    2. useState<PlayerState>('loading') for the state machine.
    3. useEffect with [src] dependency:
       - Guard: if !video || !src, return.
       - If `Hls.isSupported()`:
         - new Hls({ liveSyncDurationCount: 3, liveMaxLatencyDurationCount: 5, lowLatencyMode: false, maxBufferLength: 30, enableWorker: true })
         - hls.attachMedia(video) BEFORE hls.loadSource(src) — ordering matters per hls.js docs.
         - hls.on(Hls.Events.MANIFEST_PARSED, () => setPlayerState('playing'))
         - hls.on(Hls.Events.ERROR, (_e, data) => { if (data.fatal) { if (data.type === Hls.ErrorTypes.NETWORK_ERROR) hls.startLoad(); else setPlayerState('error'); } })
         - Cleanup: `return () => hls.destroy();` — MANDATORY, the cleanup avoids dangling MediaSource attachments on route change.
       - else if `video.canPlayType('application/vnd.apple.mpegurl')` is truthy (Safari):
         - `video.src = src`
         - `video.addEventListener('loadedmetadata', () => setPlayerState('playing'), { once: true });`
         - No hls.js instance needed — Safari uses its native HLS engine.
       - else setPlayerState('error') — unsupported browser.
    4. Render markup:

       ```tsx
       <div className={`relative w-full aspect-video bg-black ${className ?? ''}`}>
         <video ref={videoRef} controls playsInline className="w-full h-full" />
         {playerState === 'playing' && (
           <div className="absolute top-2 left-2 bg-red-600 text-white text-xs font-bold px-2 py-1 rounded" role="status" aria-live="polite">
             {badgeLabel ?? 'LIVE · delayed ~5-10s'}
           </div>
         )}
         {playerState === 'loading' && (
           <div className="absolute inset-0 flex items-center justify-center text-white">Connecting to stream…</div>
         )}
         {playerState === 'error' && (
           <div className="absolute inset-0 flex items-center justify-center text-red-400">Stream unavailable.  Please try again shortly.</div>
         )}
       </div>
       ```

    Pitfall 1 (research): parent must NEVER mount HlsPlayer when `src` is not yet available (undefined, empty string, or m3u8 will 404).  HlsPlayer itself guards via `if (!video || !src) return` — but parent must also not render the component until the stream API returns `status === 'live' && mux_playback_id`.  Document this contract in a JSDoc block at the top of the file.

    Do NOT:
    - Do not add dynamic imports for hls.js — it is a client-only component and the `'use client'` directive already prevents SSR use.
    - Do not set `lowLatencyMode: true` — Mux uses `latency_mode: 'reduced'` which is standard HLS, not LL-HLS (per 03-01 summary decisions).
    - Do not hardcode `https://stream.mux.com/...` — the URL is composed by the parent.  HlsPlayer is agnostic to Mux.
